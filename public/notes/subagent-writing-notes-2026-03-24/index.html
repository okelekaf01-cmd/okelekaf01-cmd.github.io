原创随笔 AI 方法论 Agent 工程协作 体验设计

我让 subagent 参与写这篇文章，然后更理解了协作

发表于2026-03-24更新于2026-03-24
中国
随笔AI方法论Agent工程协作体验设计

我让 subagent 参与写这篇文章，然后更理解了协作

wwxdsg2026-03-242026-03-24

最近我认真玩了一次 subagent。

不是那种“哦，原来还能多开几个 agent，挺酷”的浅层新鲜感，而是更具体一点的东西：我突然感觉到，自己平时脑子里那种会同时开很多小线程、预演分工、来回切换视角的思考方式，第一次被一个工具很明确地接住了。

体验到一半的时候，我脑子里甚至冒出一句很危险的话：

好像我的思维其实和业界顶尖是一样的，只不过落后了一些。

先声明，这句是玩笑，不是我准备在博客里郑重宣布自己已经抵达了什么新境界。
但玩笑里确实有一点真实：很多时候，我们和更成熟的方法论之间，差的未必是“有没有这种思路”，而是有没有把它稳定地组织起来、拆开来、交给合适角色去推进。

而 subagent 最有意思的地方，恰好就在这里。

以前只是脑内多人会议，现在终于有了操作界面

我之前其实已经模拟过多 agent 分工的场景，也写过相关内容。那时候我最强的感受是：一旦把角色边界、上下游依赖和拍板机制说清楚，很多“开发混乱”并不会消失，但会开始变得可管理。

这次真的把 subagent 用起来之后，我的感受又往前走了一步。

它不像传统意义上的“工具增强”，更像是把脑内那些本来只在意识里高速切换的动作，强行拉到了桌面上：

  1. 哪一部分需要单独探索
  2. 哪一部分可以并行推进
  3. 哪一部分必须由主线程拍板
  4. 谁负责起草，谁负责收口，谁只提供局部视角

以前这些动作都在脑子里一团完成，所以你会觉得自己只是“在想”。
真的把 subagent 放进工作流之后，你才会发现：原来很多思考本来就带着团队协作的形状，只是以前没有人把它们做成按钮。

最让我上头的，不是效率，而是那种“思维被显性化”的感觉

我原本以为，subagent 最直观的价值会是效率。

比如一个 agent 看代码结构，一个 agent 起草文章，一个 agent 想标题，我来负责整合，听起来当然很省时间。

但真正让我觉得新奇的，并不是“我同时叫来了几个人干活”，而是我第一次很清楚地看见了自己的思维方式是怎么被拆开的。

有些人思考时是一条很深的线，一口气往下钻。
我更像是会在脑内自然拉出几条线：

  1. 一条线先去探路
  2. 一条线先写个粗稿
  3. 另一条线负责怀疑、挑刺、改口气
  4. 最后再回到一个主视角里统一语气

这听起来很像协作，其实也是一种思考习惯。

所以我才会冒出那个有点欠打、但确实带点快乐的玩笑：
难道我和业界顶尖的区别，真的只是版本落后了一点？

当然，这话不能当真。
更准确的表达应该是：有些思维习惯本来就存在，只是顶尖的人更早把它训练成了方法，而我们现在终于开始拥有一些能配合这种方法的工具。

subagent 也没有神到哪里去，它只是把“组织能力”变重要了

这里我反而很想给它降降温。

subagent 并不是你一声令下，就会凭空长出一个完美团队。
它带来的不是“免费智力翻倍”，而是更高要求的组织问题。

你很快就会发现，真正难的事情不是“开几个 agent”，而是：

  1. 任务边界切得够不够清楚
  2. 哪些事情值得并行，哪些事情不能乱拆
  3. 什么时候应该等它，什么时候不该被它拖住
  4. 最后谁来把不同语气、不同重点的输出合成一个完整结果

换句话说，subagent 很像把你从“亲自干活的人”往“会分工、会收口的人”轻轻推了一下。

这也是为什么它让我觉得特别像真实协作。
真正的价值不只是多了几个帮手，而是你被迫开始面对协作本身：交接、边界、节奏、依赖、收敛。

这些东西，以前在人类团队里就一直重要，只不过现在它们提前出现在了你和工具之间。

这篇文章本身，就是一个很好的例子

更好玩的是，这篇文章本身就不是完全由我一个人线性写完的。

我真的把起草任务丢给了子代理。
第一位子代理甚至还很真实地给我上了一课，直接 503 Service Unavailable，下线得相当有职业精神。那一刻我居然没有觉得出戏，反而觉得这个体验更完整了：原来连“协作中有人先掉线”这种事，它都替我演到了。

后面我换了两路继续起草：一路想标题和结构，一路负责吐首稿；然后我再回来删掉不属于我的句子、修掉太满的表达、把真正想说的部分留下。

这个过程特别像一个很小的团队：

  1. 有人先搭脚手架
  2. 有人只负责提出更好的开场方式
  3. 最后还是要有人把声音统一起来

所以如果你问我，subagent 最迷人的地方是什么，我现在的答案不是“它很强”，而是：

它让很多原本只存在于脑内的协作动作，第一次有了可以被调用、被观察、被复盘的形状。

我现在对它最真实的判断

如果只把 subagent 当作“更高级一点的自动补全”，那它当然也能用，但会浪费掉它最好玩的部分。

我现在更愿意把它理解成一种新的工作界面：

  1. 它不是替你思考
  2. 它也不是替你负责
  3. 它更像是在帮你把思考拆成几个可以并行处理的角色

最后拍板的人还是你。
真正构成作品的人也还是你。
但中间那段原本混在一起的思考、试探、试写、怀疑和整合，终于不需要永远挤在一个线程里完成了。

这件事对我来说，比“效率提升了多少”更有意思。

因为它让我第一次很具体地感觉到：
我们以为自己在寻找更聪明的工具，结果有时候真正被照亮的，反而是自己原本就有、只是一直没被很好利用的思维结构。

结语

所以这次体验之后，我对 subagent 最喜欢的一点，不是它多厉害，而是它让我看见了两件事：

  1. 协作并不总是发生在团队里，它也可能早就发生在一个人的脑子里
  2. 很多“高级方法”并没有神秘到遥不可及，它们往往只是被更早、更稳定地组织起来了

至于那句“我和业界顶尖可能只是版本号差了一点”的想法，我还是坚持把它放在玩笑区。
但我会悄悄保留那一点点开心，因为这种开心本身也说明了一件事：

有时候你不是突然变厉害了。
你只是第一次遇到了一个，刚好适合你思维形状的工具。
